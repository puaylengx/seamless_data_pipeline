ETL Pattern ทั่วไป

ทุก DAG ใน project นี้ใช้ pattern เดียวกัน โดยส่งข้อมูลระหว่าง task ผ่าน Parquet file บน shared volume (./dags) แทนการส่ง DataFrame ตรง ๆ ผ่าน XCom ซึ่งมีข้อจำกัดด้านขนาด

mark_start
→
extract_to_file
→
transform
→
validate
→
load
→
notify
Taskหน้าที่Input → Output
mark_startบันทึก timestamp เริ่มต้น→ float (Unix timestamp)
extract_to_fileดึงข้อมูลจาก source DB เขียนเป็น Parquet→ str (path ของ raw.parquet)
transformทำความสะอาด, normalize, type castingraw.parquet → clean.parquet
validateตรวจสอบ schema, null, ค่าผิดปกติclean.parquet → dict report
loadโหลดข้อมูลเข้า destination (MSSQL / BigQuery)clean.parquet + report → dict stats
notifyส่ง email สรุปผล (inserted / updated / duration)stats dict → email
ℹ️
การใช้ Parquet เป็น intermediate format ทำให้ task แต่ละตัว independent — ถ้า task ใดล้มเหลว Airflow จะ retry เฉพาะ task นั้นโดยไม่ต้อง extract ข้อมูลใหม่ทั้งหมด

Shared Helpers

Module กลางใน dags/src/helpers/ ที่ทุก pipeline ใช้ร่วมกัน

audit.py — write_audit_line()

บันทึก audit log สำหรับทุก pipeline action (SUMMARY, INSERT, UPDATE) เป็น JSON line file เพื่อ traceability และ debugging

emailer.py — send_summary_email()

ส่ง email สรุปผลการ run อ่าน SMTP config จาก environment variables ใน .env เป็นอันดับแรก

Field ใน Emailมาจาก
subjectpipeline name (เช่น "DGSI : Finance Invoice")
statussuccess / no_data / failed
rows_totalจำนวนแถวทั้งหมดที่ดึงได้
inserted / updatedจำนวนแถวที่ insert / update
duration_secเวลาที่ใช้ทั้งหมด (จาก mark_start timestamp)
target_tableชื่อ table ปลายทางเต็ม (schema.table หรือ project.dataset.table)
run_dateวันเวลาที่ run (timezone: Asia/Bangkok)
updated_samplesตัวอย่าง key ที่ถูก update สูงสุด 10 รายการ

logger.py

Wrapper สำหรับ structured logging ใช้ logging.getLogger("airflow.task") เพื่อให้ log ปรากฏใน Airflow UI ที่ Log tab ของแต่ละ task instance
